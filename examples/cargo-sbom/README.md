# Example: SBOM generation for a Rust package (experimental)

This example builds [`hexyl`](https://github.com/sharkdp/hexyl) and embeds two SBOM
(Software Bill of Materials) documents into the resulting package:

1. `hexyl.cdx.json`: a CycloneDX SBOM of the crate dependency graph, generated by
   [`cargo-cyclonedx`](https://github.com/CycloneDX/cyclonedx-rust-cargo) in the build script and
   dropped into `$SBOM_DIR`.
2. `rattler-build.cdx.json`: a CycloneDX 1.5 document generated by rattler-build itself,
   describing the package, its resolved host and build environments, its run dependency specs,
   and a reference to every other SBOM file shipped alongside it.

Build it with:

```bash
rattler-build build --experimental --sbom --recipe examples/cargo-sbom/recipe.yaml
```

The build script has both a Unix and a Windows branch, so it builds on Linux, macOS and Windows.

The resulting package contains:

```
info/sboms/
├── hexyl.cdx.json          # dropped into $SBOM_DIR by the build script
└── rattler-build.cdx.json  # generated by rattler-build
```

The generated document references every collected file:

```json
{
  "type": "bom",
  "url": "./hexyl.cdx.json",
  "comment": "hexyl.cdx.json was collected from the SBOM_DIR of the build"
}
```

Collected JSON files that carry a top-level `serialNumber` and `version` are referenced with a
BOM-Link (`urn:cdx:<serial>/<version>`) instead of a relative path. `cargo-cyclonedx` omits the
serial number when `SOURCE_DATE_EPOCH` is set, which rattler-build always does for
reproducibility, so this example uses the relative form.

## How it works

`$SBOM_DIR` is only set when building with `--experimental --sbom`, which this recipe assumes.
Every regular file the build script places there is copied into `info/sboms/` unchanged; the
name `rattler-build.cdx.json` is reserved for the generated document. Collected files are
treated as opaque, self-describing documents, so any SBOM tool and format works.

`cargo-cyclonedx` is packaged on conda-forge for Linux, macOS and Windows, which is why this
recipe uses it as a `build` requirement. Alternatives:

- `cargo install cargo-cyclonedx --locked` in the build script, if you would rather not depend
  on the conda-forge package.
- [`cargo sbom`](https://github.com/psastras/sbom-rs), which emits SPDX or CycloneDX from the
  lockfile.
- A language-independent scanner such as [`syft`](https://github.com/anchore/syft), useful when
  the build vendors non-Rust components as well.

See the [SBOM generation documentation](https://rattler-build.prefix.dev/latest/experimental_features/#sbom-generation)
for the full contract.
